## 2. 测试分层策略

### 2.1 单元测试（Unit Test）— Service 层

- **注解**：`@ExtendWith(MockitoExtension.class)`
- **隔离方式**：所有 Mapper/Repository 依赖使用 `@Mock`；被测 Service 使用 `@InjectMocks`
- **覆盖重点**：业务规则、状态机流转、异常分支、边界值
- **不需要**：Spring 上下文、数据库、Redis

```java
@ExtendWith(MockitoExtension.class)
class ReportServiceTest {
    @Mock  ReportRecordMapper reportRecordMapper;
    @Mock  OperationMapper operationMapper;
    @Mock  WorkOrderMapper workOrderMapper;
    @Mock  ApplicationEventPublisher eventPublisher;
    @InjectMocks ReportService reportService;
    // ...
}
```

### 2.2 Controller 层测试（Web Layer Test）

- **注解**：`@WebMvcTest(XxxController.class)` + `@ActiveProfiles("test")`
- **隔离方式**：所有 Service 依赖使用 `@MockBean`；Security 层按需配置
- **覆盖重点**：HTTP 状态码、请求参数校验（@Valid）、响应体结构、权限控制（403/401）

```java
@WebMvcTest(AdminWorkOrderController.class)
@ActiveProfiles("test")
class AdminWorkOrderControllerTest {
    @Autowired MockMvc mockMvc;
    @MockBean  WorkOrderService workOrderService;
    // ...
}
```

### 2.3 JWT 安全测试辅助

Controller 测试中使用 Spring Security Test 提供的 `@WithMockUser` 或自定义 `SecurityMockMvcRequestPostProcessors`：

```java
mockMvc.perform(get("/api/admin/work-orders")
    .with(user("admin").roles("ADMIN")))
    .andExpect(status().isOk());
```

## 3. 测试命名与组织规范

- 测试方法命名格式：`should_[期望结果]_when_[场景条件]`
  - 示例：`should_throwException_when_orderNumberDuplicated()`
  - 示例：`should_returnToken_when_validCredentials()`
- 每个测试方法只验证一个行为
- 使用 AAA 结构：**Arrange**（准备数据）/ **Act**（执行方法）/ **Assert**（断言结果）
- 异常测试统一使用 `assertThrows(BusinessException.class, () -> ...)`

## 5. 测试覆盖率要求

| 层级 | 最低行覆盖率 | 最低分支覆盖率 | 说明 |
|---|---|---|---|
| Service 层 | **80%** | **75%** | 核心业务逻辑，必须严格测试 |
| Controller 层 | **70%** | — | 重点验证请求校验、权限控制、HTTP 状态码 |
| 整体项目 | **70%** | — | 排除 entity、dto、constant 类 |

**特别关注**（必须达到 ≥90% 行覆盖）：

- `ReportService`（报工核心）
- `WorkOrderService`（工单生命周期）
- `InspectionService`（质检状态机）
- `JwtTokenProvider`（安全核心）

**可豁免不测试的类**（无业务逻辑）：

- `entity/` 下的实体类（Lombok 生成代码）
- `dto/` 下的纯数据类
- `constant/` 下的常量类
- `WorkOrderApplication.java` 启动类

## 6. 功能完成定义 (Definition of Done)

一个功能点被视为**开发完成**，必须同时满足以下所有条件：

### 6.1 代码实现

- [ ] 功能代码已按模块结构规范实现
- [ ] 代码通过编译，无 Warning（Lombok/MapStruct 注解处理器警告除外）
- [ ] 无明显代码坏味道（重复代码、过长方法等）
